## 前言

关于 C++头文件的相关知识，曾经困扰了我好久，本文将记录我学习 C++头文件相关知识的笔记和参考资料，希望本文对您有所帮助。谢谢。

## 什么是头文件？

头文件是包含程序中使用或导入的各种函数、变量、类、宏和其他程序元素的声明的文件，通常以 `.h` 或者 `.hpp` 为后缀。若要在 C++程序中使用头文件，需要将其包含在 `#include` 预处理器指令中，该指令告诉编译器将头文件的内容复制到代码文件中。

预编译指令 `#include` 的作用是将所包含的文件全文复制到 `#include` 的位置，相当于是个展开为一个文件的宏。例如，您可以使用以下语法包含提供输入和输出函数的 `iostream` 头文件：

```c++
#include <iostream>
```

C++有两种类型的头文件：预先存在的头文件和用户定义的头文件。预先存在的头文件是 C++编译器或标准库中已经可用的文件，例如：`iostream` 、`vector` 、 `string` 等。用户定义的头文件是由程序员为特定目的（如定义类或函数）创建的文件。若要包含用户定义的头文件，需要指定文件名和路径，例如：

```c++
#include "myheader.h"
```

## 为什么要使用头文件？

简言之，就是为了美观和规范，同时便于管理代码。对于一个工程，通常将一个项目分成不同模块，针对不同模块写不同的代码，将函数声明、类定义、相关变量等内容写进 `.h` 文件，将函数定义等其他内容写进 `.cpp` (头文件和源文件内要写什么内容下面会讲到)，这样我们只需在主程序中使用 `#include` 引用不同模块的头文件 `.h` 就可以实现相关功能了。

## 头文件、对应源文件和主程序三者之间的关系？

如上所说，在每个模块对应的文件夹下，我们应该编写与该模块功能相关的头文件和源文件，通常每一个 `.cpp` 文件都有一个对应的 `.h` 文件，但也有一些例外，如单元测试文件和只包含 `main()` 函数的 `.cpp` 文件。例如一个工程有如下目录结构：（其中的 `/` 仅用于区分文件夹）

```
        project/
           |
           |-->main.cpp             (主程序)
           |-->src/                 (放置源代码)
           |-->include/             (放置头文件)
           |-->doc/                 (一些说明文档)
           |-->...                  (一些其他的东西，如构建工具、版权声明等)
```

我们可以在 src 中放置各个模块的源代码，src 目录内容如下：

```
        src/
         |
         |-->feature_one.cpp
         |-->feature_two.cpp
         |-->...
```

在 include 文件夹中放置相应源代码的头文件，include 目录内容如下：

```
        include/
            |
            |-->feature_one.h
            |-->featurn_two.h
            |-->...
```

这样我们可以在主程序 `main.cpp` 中 `#include` 相应的头文件就可以了，这里需要注意一点，我们只需 `#include` 相应的 `.h` 头文件，不需要包括对应的 `.cpp` 文件,因为这部分工作是属于项目构建工具的（如 CMake）。另外需要注意的是：这样的代码是无法直接通过 `main.cpp` 运行的，需要借助项目构建工具生成相应的 `.exe` 可执行文件，才能运行。

## 头文件和源文件内应该写些什么？

头文件中应该只放变量和函数的声明，而不能放它们的定义。不过这样的规则有三个例外：

- 头文件中可以写 `const` 对象的定义。
  因为全局的 `const` 对象默认是没有 `extern` 的声明的，所以它只在当前文件中有效。同理，static 对象的定义也可以放进头文件。

- 头文件中可以写内联函数（`inline`）的定义。
  C++ 规定，内联函数可以在程序中定义多次，只要内联函数在一个 `.cpp` 文件中只出现一次，并且在所有的 `.cpp` 文件中，这个内联函数的定义是一样的，就能通过编译。那么显然，把内联函数的定义放进一个头文件中是非常明智的做法。

- 头文件中可以写类（`class`）的定义。
  编译器只有在这个类的定义完全可见的情况下，才能知道这个类的对象应该如何布局，所以，关于类的定义的要求，跟内联函数是基本一样的。我们可以把类的定义放在头文件中，而把函数成员的实现代码放在一个 `.cpp` 文件中。也可以直接把函数成员的实现代码也写进类定义里面。在 C++ 的类中，如果函数成员在类的定义体中被定义，那么编译器会视这个函数为内联的。因此，把函数成员的定义写进类定义体，一起放进头文件中，是合法的。

- 总之，声明相关的一般写在头文件中 `.h` ，定义相关的一般写在源文件中 `.cpp` 。

## #define 保护

设想一下，如果 `a.h` 中含有类 A 的定义，`b.h` 中含有类 B 的定义，由于类 B 的定义依赖了类 A，所以 `b.h` 中也 #include 了`a.h`。现在有一个源文件，它同时用到了类 A 和类 B，于是程序员在这个源文件中既把 `a.h` 包含进来了，也把 `b.h` 包含进来了。这时，问题就来了：类 A 的定义在这个源文件中出现了两次！于是整个程序就不能通过编译了。因为 C++允许多次声明，但只允许一次实现。比如 `int foo()` ;就是一次声明，而 `int foo(){}` 就是一次实现。

这时候就需要#define 保护策略了，我们可以通过 `#ifdef` 、`#ifndef` 、`#endif` 、`#else` 等预编译指令来控制编译过程,所有头文件都应该有 `#define` 保护来防止头文件被多重包含，命名格式当是：`<PROJECT>_<PATH>_<FILE>_H_` 。

为保证唯一性，头文件的命名应该基于所在项目源代码树的全路径。例如，项目 `foo` 中的头文件 `foo/src/bar/baz.h` 可按如下方式保护：

```
#ifndef FOO_BAR_BAZ_H_
#define FOO_BAR_BAZ_H_
...
#endif // FOO_BAR_BAZ_H_
```

另外也可以使用 `#pragma once` 进行 define 保护。该命令由编译器提供保证：同一个文件不会被编译多次。注意这里所说的“同一个文件”是指物理上的一个文件，而不是指内容相同的两个文件。带来的好处是，你不必再费劲想个宏名了，当然也就不会出现宏名碰撞引发的奇怪问题。对应的缺点就是如果某个头文件有多份拷贝，本方法不能保证他们不被重复包含。但是 `#ifndef` 受语言天生的支持，不受编译器的任何限制；而 `#pragma once` 方式却不受一些较老版本的编译器支持，换言之，它的兼容性不够好。

## #include 路径及顺序

使用标准的头文件包含顺序可增强可读性，避免隐藏依赖。

推荐的 `#include` 顺序为：相关头文件，C 库，C++ 库，其他库的 `.h`，本项目内的 `.h`。

项目内头文件应按照项目源代码目录树结构排列，避免使用 UNIX 特殊的快捷目录 `.`（当前目录）或 `..`（上级目录）。
